# 总结

## 核心原则：尽量避免在线程程序中使用信号

原因：信号是进程级的概念，而线程是进程内的执行流，两者结合会产生复杂的交互和难以调试的问题

## 多线程环境中信号的传递规则

- 同步信号（如 SIGSEGV, SIGFPE）：由引发信号的线程处理（该线程本身）
- 异步信号（如 SIGINT, SIGTERM）：发送给进程，由任意一个没有阻塞该信号的线程处理
- 定时器信号（如 SIGALRM）：由进程整体处理（同异步信号）
- 使用 pthread_kill() 发送的信号：发送给指定线程

## 每个线程独立的信号掩码

1.概念：每个线程有自己的信号掩码，决定哪些信号被该线程阻塞
2.继承规则：新创建的线程继承创建者的信号掩码
3.函数：用于设置当前线程的信号掩码

```c
int pthread_sigmask(int how, const sigset_t *set, sigset_t *oldset);
```

## 共享的信号处理函数

1.概念：信号处理函数是进程级的，所有线程共享
2.设置方式：使用`sigaction()`或`signal()`，由进程的任意线程设置后，对所有线程生效
3.危险：信号处理函数是异步进行的，可能任何线程被调用，导致竟态条件和死锁

## 推荐的信号处理模式：专用的信号处理线程

1.模式步骤：

- 所有线程（包括主线程）都阻塞所有需要处理的异步信号
- 创建一个专门的线程，专门用于处理信号
- 这个专用线程调用`sigwait()`（或`sigwaitinfo()`、`sigtimedwait()`）来同步的接收信号
- 接受信号后，线程可以：
在互斥量保护下安全的修改共享数据；调用任何函数（包括非异步信号安全的函数），因为它是在正常线程上下文中执行，而非异步信号处理函数；执行复杂的清理逻辑

2.函数

```c
int sigwait(const sigset_t *set, int *sig);
```

原子性地解除这些信号的阻塞并等待，直到其中一个信号到达。返回后，该信号仍处于阻塞状态

## LinuxThreads

1.历史地位：Linux 早期的标准线程实现
2.核心问题：不完全符合 SUSv3 标准
3.主要缺陷：

- 线程管理：使用 clone() 创建线程，但管理复杂
- 信号处理：不符合 POSIX 对信号与线程交互的要求
- PID 问题：每个线程在系统中有不同的 PID，导致许多系统调用（如 getpid()）行为不符合预期
- 同步问题：某些同步原语实现效率低
- “托管线程”模型：主线程承担过多管理职责，如果主线程提前退出，整个进程可能崩溃

## NPTL

1.设计目标：完全符合 SUSv3 标准
2.核心特点：

- 1:1线程模型：每个用户线程对应一个内核调度实体
- 性能优化：高效的同步原语
- 标准符合性：完全遵循 POSIX 线程规范
- 改进的信号处理：正确实现 POSIX 对信号与线程交互的要求
- PID 统一：所有线程共享同一个 PID（getpid() 返回相同值）。
- 更好的可扩展性：支持大量线程
