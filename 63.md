# I/O模型
## 四种I/O通知模型

| 模型            | 适用场景    | 水平/边缘 | 可扩展性 |
| ------------- | ------- | ----- | ---- |
| select / poll | 少量文件描述符 | 水平触发  | 差    |
| 信号驱动I/O       | 大量文件描述符 | 边缘触发  | 好    |
| epoll         | 大量文件描述符 | 水平/边缘 | 最好   |
## `select()`与`poll()`：传统I/O多路复用
1. 工作原理
- `select()`：传入三个读/写/异常文件描述符集合，内核修改这些集合以指示哪些文件描述符已就绪
- `poll()`：传入一个`struct pollfd`数组，内核修改其中的`revents`字段以指示事件类型
2. 性能瓶颈
- 每次调用都必须传递完整的文件描述符列表给内核
- 内核需要O(N)时间遍历所有的文件描述符检查状态
- 返回后，应用程序需要O(N)时间遍历所有文件描述符以找出哪些就绪
**不具备“状态记忆”能力，内核无法记住进程正在监视的列表**
## 信号驱动I/O
1. 工作原理
- 进程为`SIGIO`信号安装处理函数
- 设定接受信号的属主进程
- 设置`O_ASYNC`标志使能信号驱动I/O
- 当文件描述符就绪时，内核发送`SIGIO`信号
- 信号处理函数→ 执行 I/O
2. 优缺点

| 优点                                        | 缺点                                        |
| ----------------------------------------- | ----------------------------------------- |
| 高扩展性：监视大量描述符时性能显著优于`select/poll`          | 信号处理复杂：传统信号不排队，可能丢失                       |
| 内核“记住”监视列表（`fcntl(F_SETOWN)` + `O_ASYNC`） | 信号处理函数可调用的函数受限（只能调用 async-signal-safe 函数） |
| Linux 增强：允许修改通知信号（如使用实时信号实现排队）            | 事件类型区分能力较弱                                |
3. Linux增强
- 实时信号排队：可将信号类型改为实时信号，多个就绪通知可排队，避免丢失
- `siginfo_t` 参数：通过 `sigaction()` 的 SA_SIGINFO 标志，可在处理函数中获取：产生信号的文件描述符、发生的事件类型（POLL_IN/POLL_OUT 等）
## epoll：Linux专有高性能I/O
1. 工作原理
- `epoll_create()`：创建 epoll 实例，返回文件描述符
- `epoll_ctl()`：添加/修改/删除需要监视的文件描述符
    - 内核记住完整的监视列表
- `epoll_wait()`：等待事件发生，返回就绪的描述符列表
2. 性能优势：
- 内核记忆：监视列表只需向内核注册一次
- `O(1)`复杂度：`epoll_wait()`仅返回就绪描述符，无需遍历监视列表
3. 事件通知模式

| 水平触发（默认） | 只要文件描述符可以I/O，就返回就绪 |
| -------- | ------------------ |
| 边缘触发     | 今当有新的I/O事件发生时通知一次  |
边缘触发要点：
- 收到通知后尽可能多的读写（循环）
- 文件描述符必须设为非阻塞
- 若未完全处理，下一次可能收不到通知
