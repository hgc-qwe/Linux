# Socket 服务器的设计模式与实现
## 迭代型服务器
1. 工作原理
- 一次只处理一个客户端
- 必须完全处理完当前客户端的所有请求后，才能接受下一个客户端
- 使用单个进程/线程循环处理
2. 典型流程
```
创建 Socket → bind → listen
    ↓
while (1) {
    accept()           // 等待客户端连接
    处理客户端请求     // 在此期间无法服务其他客户端
    close()
}
```
3. 适用场景
- 每个客户端请求处理时间极短
- 客户端数量很少且可预测
- 对相应时间要求不高的内部工具
## 并发型服务器
1. 工作原理
- 同时处理多个客户端请求
- 每个客户端由独立的子进程或线程处理
- 主进程/线程只负责接受连接，然后派生子进程/线程
2. 典型流程
```
创建 Socket → bind → listen
    ↓
while (1) {
    accept()                    // 接受客户端连接
    fork()                      // 创建子进程
    if (子进程) {
        close(监听Socket)
        处理客户端请求          // 子进程处理
        close(连接Socket)
        exit()
    } else (父进程) {
        close(连接Socket)       // 父进程关闭连接Socket
        // 继续 accept
    }
}
```
3. 适用场景
- 中等并发量的服务器
- 每个请求处理时间较长或不确定
- 客户端数量适中（几十到几百）
## 高并发场景下的设计选择
1. 性能瓶颈
- 进程/线程创建开销
- 上下文切换开销：大量进程/线程导致调度开销大
- 内存占用
- 文件描述符限制：系统可打开的`socket`数量有限
1. 高并发服务器的设计方法

| 方法      | 描述                    |
| ------- | --------------------- |
| I/O多路复用 | 单线程监视多个`socket`，就绪时处理 |
| 线程池     | 预先创建固定数量的线程，循环处理任务    |
| 进程池     | 预先创建固定数量的子进程          |
| 事件驱动    | 异步I/O + 回调机制          |
| 协程      | 用户态轻量级线程              |
## `inetd`超级服务器
1. 定义
- `Internet`超级服务器守护进程
- 可以监视多个`socket`
- 当请求达到时，启动合适的服务器进程来处理
2. 工作流程
```
1. inetd 启动，读取配置文件 /etc/inetd.conf
2. 为每个服务创建 socket 并监听
3. 使用 select() 监视所有 socket
4. 当某个 socket 有请求时：
   - fork() 子进程
   - 将 socket 复制到标准输入/输出（stdin/stdout）
   - exec() 对应的服务器程序
5. 子进程处理请求，完成后退出
```
